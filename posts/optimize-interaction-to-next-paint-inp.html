How to Optimize Interaction to Next Paint (INP)

Alex Zen
• July 11, 2026 • 5 min read

Understanding the Shift to Interaction to Next Paint (INP)

In March 2024, Google officially elevated Interaction to Next Paint (INP) to a Core Web Vital, completely replacing First Input Delay (FID). While FID only measured the delay of the very first interaction on a page, INP evaluates a page's overall responsiveness throughout its entire lifecycle by tracking the latency of all click, tap, and keyboard interactions. This paradigm shift means web performance is no longer just about a fast initial load; pages must remain fluid and responsive as users deeply interact with dynamic features, toggle state changes, and submit complex forms. For developers, designers, and SEO professionals, mastering INP is the key to maintaining organic search dominance and delivering friction-free user experiences.

The Three Phases of Page Latency

To systematically optimize INP, we must break down a user interaction into its three distinct component phases. First is the Input Delay, which is the time between when a user initiates an action (e.g., clicking a button) and when the corresponding event handlers begin to execute. This is typically caused by a busy main thread occupied with heavy background JavaScript tasks. Second is the Processing Time, which represents the execution time of the JavaScript event listeners themselves. Finally, there is the Presentation Delay, which is the time required for the browser to recalculate styles, perform layout changes (reflow), repaint the screen, and physically composite the new frames to display visual feedback to the user. Achieving a "Good" INP score of 200 milliseconds or less requires optimizing all three phases.

"Speed is no longer just a technical metric; it is the foundational layer of digital empathy and user trust. When a website responds instantly, it respects the user's time and attention."

Phase 1: Minimizing Input Delay by Yielding to the Main Thread

When the main thread is blocked by a long-running JavaScript task (any task exceeding 50ms), user interactions are queued and must wait, resulting in high Input Delay. To solve this, developers must break up long tasks into smaller, asynchronous chunks, allowing the browser to "yield" to high-priority rendering and user input tasks. The modern scheduler.yield() API is the gold standard for this, though we can implement an elegant fallback using microtasks or setTimeout for broader browser compatibility.

// A utility function to yield execution back to the main thread
function yieldToMain() {
  if (globalThis.scheduler && typeof globalThis.scheduler.yield === 'function') {
    return scheduler.yield();
  }
  return new Promise(resolve => setTimeout(resolve, 0));
}

// Example of breaking up a heavy data-processing loop
async function processLargeDataSet(items) {
  for (let i = 0; i < items.length; i++) {
    // Process an item
    processItem(items[i]);

    // Yield to the main thread every 50 iterations to keep the UI responsive
    if (i % 50 === 0) {
      await yieldToMain();
    }
  }
}

Phase 2: Optimizing Processing Time with Non-Blocking Event Handlers

Processing Time can escalate rapidly if event handlers attempt to do too much synchronous work. A common pitfall is executing heavy state mutations, telemetry tracking, and UI updates in the exact same execution block. To optimize this, developers should decouple immediate visual feedback (like closing a modal or showing a loading spinner) from non-critical business logic or analytics logging. Non-urgent tasks can be deferred using requestIdleCallback(), while critical visual updates should be aligned with the browser's natural rendering cycle.

const filterButton = document.querySelector('#apply-filters');

filterButton.addEventListener('click', (event) => {
  // 1. Immediate UI Feedback (High Priority)
  showLoadingSpinner();

  // 2. Defer heavy data filtering to the next frame
  requestAnimationFrame(() => {
    const results = runHeavyFilterLogic();
    renderResults(results);
    hideLoadingSpinner();
  });

  // 3. Offload non-critical telemetry to idle periods
  if ('requestIdleCallback' in window) {
    requestIdleCallback(() => {
      sendAnalyticsData({ interaction: 'apply-filters' });
    });
  } else {
    setTimeout(() => {
      sendAnalyticsData({ interaction: 'apply-filters' });
    }, 200);
  }
});

Phase 3: Reducing Presentation Delay and Layout Thrashing

The presentation delay phase is heavily impacted by the DOM's layout engine. When event handlers modify the DOM, they can trigger "forced synchronous layouts" or "layout thrashing"—an anti-pattern where JavaScript reads layout properties (such as element.offsetHeight or getBoundingClientRect()) immediately after modifying styles. This forces the browser to calculate the layout ahead of schedule. To prevent this, always batch your DOM reads first, and then batch your DOM writes. Additionally, utilize modern CSS properties like content-visibility: auto to skip rendering off-screen elements, and keep your overall DOM depth and complexity minimal to speed up recalculation times.

Conclusion: Continuous Monitoring and Refinement

Optimizing for INP shifts our focus from one-off, synthetic load-time performance to continuous, real-world runtime fluidity. While synthetic laboratory tools like Lighthouse are excellent for spotting structural regressions, real-user monitoring (RUM) tools and field data via the Chrome User Experience Report (CrUX) are essential for identifying the precise, chaotic interactions real users experience on mobile devices and slower networks. By prioritizing thread yielding, optimizing event handler pathways, and eliminating layout bottlenecks, engineering teams can deliver snappy, delightful user interfaces that satisfy search algorithms and human users alike.
