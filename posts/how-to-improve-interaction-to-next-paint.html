How to Improve Interaction to Next Paint

Alex Zen
• July 13, 2026 • 5 min read

The New Gold Standard of Web Responsiveness: Mastering INP

In March 2024, Google officially replaced First Input Delay (FID) with Interaction to Next Paint (INP) as a Core Web Vital. While FID only measured the delay of the very first user interaction, INP evaluates the latency of all clicks, taps, and keyboard interactions throughout the entire lifespan of a page. For professional web developers, designers, and SEO specialists, this change represents a paradigm shift. We can no longer rely on optimizing only the initial load; we must ensure that our applications remain fluid, responsive, and performant during prolonged user sessions. Improving your INP score directly translates to lower bounce rates, higher conversion rates, and better search visibility.

Deconstructing the Interaction Lifecycle

To optimize INP, we must first understand what occurs between the moment a user interacts with the page and the moment the browser paints the next frame. This timeline is divided into three distinct phases:

  • Input Delay: The time between when the user initiates an interaction and when the event handler callbacks begin to execute. This is primarily caused by main thread congestion from background tasks.
  • Processing Time: The total time required to execute all event handlers associated with the user interaction.
  • Presentation Delay: The time it takes for the browser to recalculate styles, lay out the page, repaint the pixels, and composit the frame onto the screen.

By identifying which of these three pillars is driving up your latency, you can apply targeted optimization strategies to dramatically reduce your interaction overhead.

Strategic Technical Interventions for Low-Latency Experiences

1. Yielding Long Tasks to the Main Thread

When JavaScript executes on the main thread, it blocks the browser from responding to user inputs. Any task taking longer than 50 milliseconds is considered a "long task" and is a primary culprit for high Input Delay. To keep the main thread free, developers must break up complex, long-running processes into smaller, asynchronous chunks. The modern way to handle this is via the scheduler.yield() API, with a fallback to older microtask/macrotask yielding strategies for older browsers.

// A utility function to yield execution back to the main thread
async function yieldToMain() {
  if ('scheduler' in window && 'yield' in window.scheduler) {
    await scheduler.yield();
  } else {
    // Fallback for browsers that do not support scheduler.yield
    return new Promise(resolve => setTimeout(resolve, 0));
  }
}

// Example of processing a large array without blocking the UI
async function processLargeDataset(items) {
  for (let i = 0; i < items.length; i++) {
    processItem(items[i]);
    
    // Yield to the main thread every 50 items to keep the page responsive
    if (i % 50 === 0) {
      await yieldToMain();
    }
  }
}

2. Eliminating Layout Thrashing in Event Handlers

Layout thrashing occurs when JavaScript repeatedly writes to the DOM and then immediately reads geometric properties (like offsetHeight or getBoundingClientRect()) within the same frame. This forces the browser to synchronously recalculate the style and layout, causing a massive spike in Processing Time. To prevent this, developers should always batch DOM reads first, followed by DOM writes, or leverage the requestAnimationFrame API to schedule visual updates gracefully.

// BAD: Causes layout thrashing by reading and writing in a tight loop
function resizeCardsBad(cards) {
  cards.forEach(card => {
    const width = card.parentElement.offsetWidth; // Read
    card.style.width = `${width}px`;             // Write (forces layout recalculation)
  });
}

// GOOD: Batches reads together, then schedules writes
function resizeCardsGood(cards) {
  // 1. Batch all reads
  const widths = cards.map(card => card.parentElement.offsetWidth);

  // 2. Schedule all writes in the next animation frame
  requestAnimationFrame(() => {
    cards.forEach((card, index) => {
      card.style.width = `${widths[index]}px`;
    });
  });
}

3. Minimizing Presentation Delay via CSS and DOM Optimization

Even if your JavaScript runs in under 10 milliseconds, your INP score can still suffer if the browser takes 300 milliseconds to render the resulting visual changes. To minimize Presentation Delay, designers and developers must collaborate on DOM health. Keep your overall DOM depth shallow and limit the number of nodes. Additionally, make strategic use of CSS properties like content-visibility: auto to skip rendering off-screen elements, and ensure that animations are handled by compositor-only properties such as transform and opacity rather than layout-triggering properties like top, left, or height.

Continuous Monitoring and the Path Forward

Achieving a "Good" INP score (under 200 milliseconds) is not a one-time task; it requires ongoing vigilance and integrated monitoring workflows. Synthetic lab testing tools, such as Lighthouse and PageSpeed Insights, are great for initial diagnoses, but they cannot replicate real-world user behavior. To truly master INP, you must implement Real User Monitoring (RUM) using the web-vitals JavaScript library to capture actual field data from your production users. By continuously analyzing interaction telemetry across different devices and network speeds, your team can catch responsiveness regressions before they impact your conversion rates and organic search rankings.
